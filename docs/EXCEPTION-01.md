# EXCEPTION-01：セッションロック中の例外操作制御（確定）

論点ID：EXCEPTION-01  
ステータス：確定（再オープン禁止）  
構成：#018（配送業者変更）・#030（保留設定）  

---

## 設計範囲

- セッションロック中に発生しうる例外操作の定義
- 各操作の許可可否および処理方法
- BASE再取得要否との関係

---

## 共通設計原則

BASE側注文データの変更を伴わない操作は、セッションロック中でも実行可能である。

これらの操作はUpstash上で完結し、BASE再取得を必要としない。セッションロック状態および注文集合は維持される。

| 原則 | 内容 |
|---|---|
| 実行可能条件 | BASE側注文データの変更を伴わないこと |
| 処理の完結場所 | アプリ内（Upstash）のみ |
| BASE再取得 | 不要 |
| セッションロック状態 | 維持する |
| 注文集合 | 変更しない |

---

## #018：ロック中の配送業者変更（確定）

### 確定内容

セッションロック中の配送業者変更は汎用操作として扱う。

配送業者変更はアプリ内（Upstash）で完結する操作であり、BASE側注文データの変更を伴わない。このためBASE再取得は不要であり、セッションロック状態および注文集合に影響しない。

| 項目 | 内容 |
|---|---|
| 操作の性質 | 汎用操作（特定ケースへの限定なし） |
| 代表例 | e飛伝で離島判明時のヤマト宅急便への切替 |
| 操作の完結場所 | アプリ内（Upstash）のみ |
| BASE再取得 | 不要 |
| セッションロック状態 | 維持する |
| 注文集合 | 変更しない |

### 発生ケース例

| ケース | 内容 |
|---|---|
| 離島判明（代表例） | e飛伝で離島アラート発生→佐川不可→ヤマトへ切替 |
| ネコポス→佐川格上げ | 納品書確認時点でサイズ超過と判断→佐川へ変更 |
| 佐川→ヤマト振替 | その他事情による配送業者変更 |

### 既存設計との接続

概要設計書4-2の「ロック後e飛伝で離島判明した場合の例外処理フロー」は、離島ケースを代表例とする本汎用操作の一実例として位置づける。フロー構造（Upstash上の変更のみ・BASE再取得不要・セッションロック継続）は発生理由を問わず共通である。

### 概要設計書への反映状態

未反映（補強レイヤーとして保持）

## #030：保留設定の発生タイミング（確定）

### 確定内容

T8（保留設定）はセッションロック前・ロック後を問わず発生しうる。

T8はアプリ内（Upstash）で完結する操作であり、BASE側注文データの変更を伴わない。このためBASE再取得は不要であり、セッションロック状態を維持したまま実行可能である。

| 項目 | 内容 |
|---|---|
| 発生タイミング | ロック前・ロック後を問わない |
| 操作の完結場所 | アプリ内（Upstash）のみ |
| BASE再取得 | 不要 |
| セッションロック状態 | 維持する |
| 処理内容 | hold_flag ON・checklist_printed_flag OFF（ロック前後共通） |

### 状態遷移との接続

T8によりhold_flagがONになった注文はT10（出荷対象注文の変更）に接続し、出荷対象集合から除外される。T10はロック後にT8が発生した場合も同様に適用される。

| トリガー | 処理 | 接続先 |
|---|---|---|
| T8（保留設定） | hold_flag ON・checklist_printed_flag OFF | T10へ接続 |
| T10（出荷対象注文の変更） | 出荷対象集合から除外 | — |

### #031との関係

#031（ロック後キャンセルを保留フローで吸収）は、T8がロック後に発生しうる代表例である。

### 既存設計との接続

| 文書 | 接続内容 |
|---|---|
| DATA-01 T8 | 発生条件「スタッフが注文を保留に設定」はロック前後の制約を持たない。本明文化はT8の処理定義を変更するものではなく、発生タイミングの範囲を補完する |
| DATA-01 T10 | T8→T10の接続はロック後発生時も同様に適用される |
| FLOW-01 S1 | pdf_output_done_flagのOFF条件に「保留設定／解除」が明示されており、ロック後の保留発生を前提とした構造と整合する |
| 概要設計書5-1 | セッション終了条件「保留設定された注文はセッション対象から除外」はロック後の保留発生を前提とした記述であり整合する |

### 概要設計書への反映状態

未反映（補強レイヤーとして保持）
