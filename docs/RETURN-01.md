# 設計補強メモ RETURN-01

論点ID：RETURN-01 #012
タイトル：棚戻し完了の確認手段の定義
ステータス：確定済み・再オープン禁止
設計目的：棚戻し完了の保証ではなく、棚戻し指示の未確認による次工程への誤進入防止

---

## 1. 確定内容

### 棚戻し完了の定義

| 項目 | 確定内容 |
|---|---|
| 完了の定義 | 棚戻し指示リストをスタッフが確認し、「確認完了」操作を実行した時点 |
| 保証する範囲 | 棚戻し指示リストを確認したこと |
| 保証しない範囲 | 物理的に全商品を棚へ戻したこと |
| 完了単位 | 現在表示されている棚戻し指示リスト単位 |
| 部分完了 | 許容しない。指示リスト全体への確認操作が完了条件 |

### 設計の位置づけ

棚戻しは物理作業であり、システムによる完全な完了保証は不可能。本設計の目的は「棚戻し指示リストを見ずに次工程へ誤進入すること」の防止に限定する。物理作業の完了責任はスタッフの判断に委ねる。これはSAFEGUARD-01の「業務性質上の限界として許容する」範囲と同列の扱いとする。

---

## 2. 確認操作の仕様

### 操作フロー

1. 差分検知（T2またはT3）が発生する
2. 棚戻し指示リストが画面に表示される（既存設計通り）
3. スタッフが指示リストの内容を確認する
4. 「棚戻しを確認しました」ボタンを押下する
5. 次工程（再ピッキング画面または差分確認完了）へ進める

### ボタン押下前の制御

| 状態 | 制御 |
|---|---|
| 棚戻し指示リスト表示中・未確認 | 次工程への遷移を禁止 |
| 「棚戻しを確認しました」押下後 | 次工程への遷移を許可 |

※棚戻し指示リストを一定量スクロールするまで「棚戻しを確認しました」ボタンは活性化しない

### 既存設計との整合

| 確認項目 | 整合状態 |
|---|---|
| 新規フラグの追加 | なし。UIフロー上のステップとして実装。Upstashへの状態書き込みは不要 |
| DATA-01の変更 | なし。T2・T3のリセット処理は既存定義を維持 |
| 新規画面の追加 | なし。棚戻し指示リストは既存の差分確認画面内に表示 |
| SAFEGUARD-01との整合 | ✅ 物理作業の限界として許容済みの範疇に収まる |
| CONFIRM-01との整合 | ✅ 「システム保証できない領域は人間の確認に委ねる」原則に準拠 |

---

## 3. ミスパターンと防止分類（確定版）

| ミスパターン | 防止区分 | 手段 |
|---|---|---|
| 指示リストを見ずに次工程へ進む | 構造防止 | 確認ボタン押下なしに次工程遷移を禁止 |
| 一部商品の棚戻し忘れ | 注意依存（許容） | 業務性質上の限界 |
| 誤棚戻し（不要商品を戻す） | 注意依存（許容） | 業務性質上の限界 |

---

## 4. スマレジ在庫の扱い

棚戻し時のスマレジ在庫書き戻しは現フェーズのスコープ外とする。スマレジ連携は出荷完了時の書き込みのみが定義済みであり、棚戻し方向の書き込みは要件定義書・概要設計書いずれにも含まれていない。フェーズ3着手前チェックリストへの追加を推奨する。

---

## 5. 拡張候補：逆ピッキング（マイナスピッキング）

### 案の概要

岡本さんが現場フロー検証vol.1にて提案した案。棚戻しをピッキングの逆方向操作として定義し、スマホのバーコードスキャンで棚戻し完了を商品単位で確認するフロー。

### 想定効果

- 棚戻し完了の確認が商品単位の構造防止に格上げされる
- 注意依存で残っている「一部戻し忘れ」「誤棚戻し」が検知可能になる
- 棚戻し完了の記録がUpstashに残る

### 必要要素

| 必要要素 | 内容 |
|---|---|
| 新規フラグ | 棚戻し完了フラグ（商品単位またはイベント単位） |
| 新規画面 | 棚戻しスキャン画面（フェーズA内に追加） |
| 状態遷移の追加 | 棚戻しスキャン完了→再ピッキング開始への遷移定義 |
| DATA-01の再設計 | U4への棚戻し進捗フィールド追加、リセット条件の拡張 |

### 採用条件

- フェーズ3以降に着手する
- DATA-01の再設計を伴う変更として扱う
- 運用開始後の実績（棚戻しミスの発生頻度）を踏まえて必要性を再評価する
