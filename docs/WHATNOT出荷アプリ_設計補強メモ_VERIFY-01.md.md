# 設計補強メモ

論点ID：VERIFY-01
タイトル：最終確認チェックシートの確認目的・記載項目・フラグ設計
ステータス：確定済み

-----

## 0. 設計思想

チェックシートは「送り状が貼られた箱との目視照合」を唯一の目的とする書類である。
照合に不要な情報は載せない。最小限の項目で最終防御を成立させる。

-----

## 1. チェックシートの目的

- 確認対象：送り状が貼られた箱
- 照合相手：箱に貼られた送り状
- 確認行為：チェックシートと送り状の目視照合
- 目的：注文の抜け漏れを出荷最終段階で防ぐこと

チェックシートは照合後に破棄する。保管運用は行わない。

-----

## 2. 記載項目（確定）

|項目    |役割   |データソース           |照合先（送り状）               |
|------|-----|-----------------|-----------------------|
|お届け先氏名|主照合キー|order_receiver基本※|送り状「お届け先」欄             |
|お届け先住所|主照合キー|order_receiver基本※|送り状「お届け先住所」欄           |
|配送業者  |補助確認 |Upstash（配送業者選択状態）|送り状の種別（佐川／ヤマト／ネコポス）    |
|注文ID  |補助識別 |BASE API         |送り状「お客様管理番号」（視認性低・補助用途）|

※ order_receiverが存在しない／空の場合はorder_purchaserにフォールバック（注文単位）。
　 フォールバック条件・単位はDEST-01 D7に準拠。

**禁止事項**

- 上記4項目以外の追加禁止
- 注文者（purchaser）情報の追加禁止
- 商品名・数量・金額・備考の追加禁止
- receiverとpurchaserを同列の照合対象として扱うことは禁止

-----

## 3. DEST-01との整合

|項目       |チェックシート               |CSV出力（D1）     |一致|
|---------|----------------------|--------------|--|
|お届け先氏名   |receiver基本（D7準拠・住所に拡張）|receiver基本（D1）|✅ |
|お届け先住所   |receiver基本（D7準拠・住所に拡張）|receiver基本（D1）|✅ |
|フォールバック条件|注文単位でpurchaser        |注文単位でpurchaser|✅ |

- D6（メイン画面）はpurchaser表示、チェックシートはreceiver基本。これはDEST-01確定済みの意図的な非連動であり矛盾ではない。
- D7のルール（receiver基本・注文単位フォールバック）を住所フィールドに拡張して適用する。

-----

## 4. 印刷済みフラグ設計

### フラグの定義

「印刷時点のチェックシート内容と、現在の出荷対象集合・配送内容が一致している」
ことを示す状態フラグ。Upstashで管理。

### フラグON条件

チェックシートPDFの印刷操作が完了した時点でONになる。

### フラグリセット条件（内容変化ベース）

|リセット条件             |影響するチェックシート項目   |
|-------------------|----------------|
|配送業者の変更            |補助確認（配送業者）が変わる  |
|お届け先情報（receiver）の変更|主照合キー（氏名・住所）が変わる|
|出荷対象注文の変更（保留・除外含む） |対象注文集合が変わる      |

### リセットしない操作

- CSV再出力（内容変化を伴わない場合）
- 送り状番号の入力・変更
- アプリ内メモの変更

### フラグOFF時の制御

出荷完了ボタンを非活性化する。再印刷が必須。

-----

## 5. 禁止事項（全体共通）

- 記載項目の追加禁止
- 操作ベース（ボタン押下など）のフラグリセットは禁止
- 内容が変わらない限りフラグを維持する
- チェックシートを保管運用に転用しない
